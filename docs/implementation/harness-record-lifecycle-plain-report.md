# 这次记录改造，实际改了什么

先说结论：代码层面的记录改造已经收口，并通过了完整测试。真实 Obsidian 还没有
迁移、部署或清理，所以你现在正在使用的插件不会被这次开发直接改变。

这次不是简单地把“临时会话多保存几天”，而是重新分清两件事：

- 你真正拥有的聊天、成果和长期记忆，由 EchoInk 保存。
- Codex、OpenCode、Hermes 自己产生的 thread、session 和 run，只是执行时用的缓存。

这样一来，底层 Agent 会话可以按各自能力复用、归档或清理，但不会再决定你的聊天历史
还在不在。

## 1. 这次改了什么？

第一，EchoInk 现在有一套统一的正式对话记录。

无论这一轮使用 Codex、OpenCode 还是 Hermes，看到和续接的都是同一份 EchoInk 对话。
V2 版本已经可以正常新建会话、追加消息、更新上下文、启动恢复、清空记录和删除会话，
不再只能读取迁移后的数据。

第二，把“用户记录”和“Agent 临时会话”彻底分开。

Codex 的 thread、OpenCode 的 session、Hermes 的 session/run 仍然会记录，因为短期复用
可以省时间和上下文。但它们不再是唯一历史，也不会和报告、Wiki、任务结果、长期记忆
绑在一起删除。

第三，三个后端按真实能力处理，不再假装完全一样。

- Codex 可以复用 thread，清理时使用归档。
- OpenCode 可以复用和删除 session。
- Hermes 公开支持列出和加载旧 session，所以可以安全续接；目前没有公开稳定的删除
  能力，因此仍明确显示“不支持清理”，不会把关掉进程说成删除成功。

第四，把几个容易混淆的动作拆开了。

- “开启新上下文”只是从新的起点继续聊，旧历史仍在。
- “重置 Agent 缓存”只换掉底层 Agent 会话，不删 EchoInk 对话。
- “清空会话记录”会建立一个新的空对话状态，并安全退休旧正文。
- “删除会话”会先留下正式删除凭证，让旧数据不能意外复活，再等待安全清理。

第五，补上了真实迁移入口。

迁移开始时，插件会先停止接收新任务，等正在运行的任务和后台清理结束，再短暂暂停
Conversation、Settings、Raw、Run、Memory、History 和 Native Execution 的新写入。
复制、核对和切换完成后，这些入口自动恢复。中途报错也会自动解锁，不会把插件一直
卡在停写状态。

迁移不是“复制完文件就算成功”。插件会重新读取设置、任务记录和 Agent 执行记录。
数据缺失、内容变化或前后对不上时，只会返回“不能迁移”，不会强行切换。

旧数据还有一个现实问题：这些记录生成时，正式的 Run 账本还不存在。新版不会倒推
并伪造一批任务记录，而是建立一份只读兼容档案。档案只保存匿名关系摘要，并和原始
V1 数据的完整指纹绑定；原始 V1 文件也继续保留。这样既能证明旧关系没有丢，也不会
把后来补写的内容冒充成当年的真实运行记录。上线后的新任务则直接进入正式账本。

为了保证以后还能安全回退，新版记录也可以导出成旧版能读取的格式。导出时会把
“正文里本来就有的摘要”和“外部记录库提供的证明”分开核对，不会因为同一条关系
被算了两次而误报，也不会为了通过校验放宽缺失、冲突或重复关系的阻断规则。

## 2. 跟之前有哪些不一样？

| 事情 | 之前 | 现在 |
| --- | --- | --- |
| 聊天历史由谁负责 | EchoInk 记录和 Agent 原生会话边界不清 | EchoInk 对话是正式历史，Agent 会话只是缓存 |
| 切换 Codex、OpenCode、Hermes | 容易依赖各自旧会话 | 三个后端共享同一份 EchoInk 上下文 |
| V2 日常使用 | 切到 V2 后只能阻断旧写入，还不能完整工作 | V2 已能正常读写、清空、删除和恢复 |
| Hermes 连续对话 | 主要靠重建临时 session | 能安全加载时复用原 session |
| 清空与删除 | 某些路径只是隐藏消息或删一个旧 ID | 新上下文、缓存重置、清空、删除各有明确结果 |
| 后端清理 | 容易把三种后端写成同一种“已删除” | Codex 归档、OpenCode 删除、Hermes 如实显示不支持 |
| 迁移过程 | 复制期间仍可能有新写入 | 先等旧任务结束，再统一短暂停写和切换 |
| 迁移判断 | 可能相信调用方提交的清单 | 必须从真实记录库重新核对 |
| 清理失败 | 容易连带把已完成任务判成失败 | 业务结果和缓存清理分开记录 |

## 3. 之前是什么样的？现在是什么样的？

以前，EchoInk 更像一个把任务转交给 Agent 的入口。每轮运行结束后，再想办法处理
Agent 留下的 thread 或 session。因为用户对话、任务日志、Agent 缓存、报告和记忆
没有完全分开，所以无论“全部删掉”还是“全部长期保留”都不合理。

现在，EchoInk 更像一套完整的 harness，也就是统一管理三种 Agent 的外壳。

你真正需要长期保留的东西，放在 EchoInk 自己的正式记录里；底层 Agent 会话只在有用
时短期复用。它们丢失后，EchoInk 仍能用自己的对话和上下文继续工作。这样既不会因为
清理临时会话丢掉历史，也不用永久保存所有运行缓存。

代码现在已经具备安全切换到 V2 的能力。最新真实数据已经重新复制到两份全新副本做
搬家演练，19 个会话、251 条消息和 17 份上下文摘要全部迁入；526 条旧关系完成只读
归档，校验结果没有缺失或冲突。迁移前后旧 V1 的 58 个文件数量和整体指纹完全一致，
重复执行也只会报告“已经完成”，不会再搬一遍。

副本通过不等于正式上线。本轮到这里仍没有修改真实 Vault、删除历史或启动自动清理。
正式步骤是：先把这版代码安全合入你已经提交过的 `main`，部署并重载插件，再备份当时
最新的数据，通过插件自己的停写窗口执行正式搬家，最后分别用 Codex、OpenCode 和
Hermes 做真实对话验收。
