# 这次记录改造，实际改了什么

先说进度：开发分支已经完成 Phase 0–3，并提交到
`19f183b feat(harness): add Hermes ACP session resume`。这些改动还没有部署到
正在使用的 Obsidian。Phase 4 是备份、真实迁移、部署和实机验收，目前没有执行。

这次改造解决的是一个根本问题：EchoInk 既要保存用户真正需要的对话和成果，又不能
把 Codex、OpenCode、Hermes 每次运行留下的临时缓存，误当成唯一历史长期保存。

## 1. 这次改了什么？

现在，记录被分成了五类，各管各的：

- 对话记录：保存你和 AI 真正看到的聊天内容。切换 Codex、OpenCode 或 Hermes，
  仍然读同一份 EchoInk 对话。
- 任务记录：保存一次操作是否成功、由谁执行、结果放在哪里，方便以后查问题。
- Agent 临时会话：Codex thread、OpenCode session、Hermes session 只是加速缓存，
  不是聊天历史的唯一来源。
- 业务成果：报告、Wiki、Tracker、编辑器里确认过的内容独立保存，不跟着临时会话
  一起删。
- 长期记忆：正式记忆和待确认内容单独管理，不允许删除会话时顺手误删。

“开启新上下文”“重置 Agent 缓存”“清空会话记录”“删除会话”现在也是四个不同
操作。以前有些地方只是把旧消息藏起来；现在每个操作都有自己的规则，失败后也能恢复。

迁移和清理也加了安全门。插件不能再凭一份“已经处理”的清单就相信迁移完成，必须
重新读取设置、Agent 执行记录和任务记录。少一处、文件损坏、版本不认识或记录对不上，
都会停下来保留原数据。

Hermes 这次也补齐了真实能力判断。本机 Hermes 公开支持列出和加载旧 session，
开发分支现在可以安全复用同一个 session。Hermes 没有公开稳定的删除接口，所以系统
仍然明确显示“不支持清理”，不会把关闭进程假装成删除成功。

## 2. 跟之前有哪些不一样？

| | 之前 | 现在 |
| --- | --- | --- |
| 聊天历史由谁负责 | EchoInk 本地记录和 Agent 原生会话混在一起，边界不清 | EchoInk 对话是唯一真源，Agent 会话只是缓存 |
| 切换 Agent | 容易依赖各自旧 thread/session | Codex、OpenCode、Hermes 共享同一份 EchoInk 上下文 |
| Hermes 连续对话 | 每轮主要靠 EchoInk 把旧内容重新塞进去，再建新 session | Hermes 自己公开能力时复用原 session；能力缺失时才继续用旧办法 |
| 清空与开启新对话 | 有些路径只是隐藏旧消息 | 开启新上下文、清空记录、删除会话各有明确结果 |
| Agent 会话清理 | 三个后端容易被写成同一种“已删除” | Codex 是归档，OpenCode 可删除，Hermes 仍是不支持 |
| 迁移是否成功 | 可能相信调用方提交的处理清单 | 必须从真实记录库重新核对 |
| 自动清理 | 容易把“能够恢复清理事务”误解成“已经自动清理” | 当前只恢复已经开始的事务，不自动发起新删除 |
| 清理失败 | 容易把已经成功的任务一起判成失败 | 业务结果和缓存清理分开记，互不冒充 |

## 3. 之前是什么样的？现在是什么样的？

以前，插件更像一个调用入口。每次把任务交给 Agent，Agent 自己产生 thread、session
或 run；任务结束后再尝试归档或删除。EchoInk 虽然也保存消息，但对话、任务日志、
Agent 缓存、报告和记忆之间的归属没有完全分开。于是会出现两个风险：临时会话删掉后
担心历史丢失，或者为了保留历史把大量执行缓存一直留着。

现在，EchoInk 更像一套完整的 harness，也就是统一管理三种 Agent 的外壳。用户真正
拥有的对话、成果和长期记忆由 EchoInk 保存；Agent 原生会话可以短期复用，但丢失后
也能从 EchoInk 上下文重建。这样既不用依赖某一个后端的历史，也不用永久保存所有
临时运行记录。

真实数据还没有迁移，也没有清理。我们在插件数据副本上做的检查仍然主动阻断：

- 正式任务记录库目前缺失。
- 还有 517 条旧执行关系找不到新的负责记录。
- 旧会话里还有一处数据结构对不上当前迁移规则。

另外，Phase 4 审计确认：插件日常 Conversation 写入还没有完整切到 V2，也没有一个
能同时暂停新任务和全部写入的正式迁移入口。现在强行切换，可能出现“读取新记录，
却继续把新内容写回旧记录”的分叉。因此这一步不能靠补一个按钮或多跑测试解决，
必须先完成正式读写路由，再在真实数据证据齐全后迁移。

现在已经补上第一道保护：插件不再固定向旧目录写入，而是让读取和写入共同服从同一份
迁移路由。没有迁移时继续正常使用旧记录；反向恢复后使用恢复出来的新 V1 记录；
如果 V2 已经切成 active，但完整的日常写入能力还没接好，插件会明确停止，不会偷偷
把新内容写回旧记录。

所以真实迁移仍未执行，不是还有一堆测试没跑完，也不是为了追求流程完整。真正原因是：
Phase 4 会碰真实数据，而当前证据和产品接线还不允许安全执行。继续硬做只会增加风险
和无效工作。

本轮没有部署插件，没有修改真实 Vault，没有删除历史，也没有启动自动 retention、
Raw 清理或 Native 批量清理。下一步如果要进入真实迁移和 Obsidian 验收，需要先单独
补齐 V2 的日常清空/删除和迁移静默窗口，再确认备份、迁移和部署。
