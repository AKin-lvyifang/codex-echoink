# Harness 记录改造：这次到底改了什么

这次解决的是一个很具体的问题：插件准备搬迁或回退会话记录时，不能只看“文件好像
复制过去了”，还必须确认那些没有放在会话正文里的执行记录，确实由对应的地方接住了。

## 1. 这次改了什么？

我给迁移过程加了一道真实核对。

现在，插件必须亲自去读取三处记录：

- 设置文件：保存会话的设置和当前状态。
- Agent 执行记录：保存 Codex、OpenCode、Hermes 这类后端实际创建过的执行对象。
- 任务运行记录：保存一次调用属于哪个任务、是否完成、用了哪个后端。

只有这三处真实记录能够和旧会话对上，迁移才允许继续。少一处、文件损坏、版本看不懂，
或者记录对不上，都会停下来。

这套检查同时接到了“迁到新版记录”和“从新版导出兼容旧版”两条路上。也就是说，向前
迁移和向后回退都不能绕过它。

## 2. 跟之前有哪些不一样？

| | 之前 | 现在 |
| --- | --- | --- |
| 怎么证明记录已经接住 | 调用迁移功能的人可以直接提交一份“已经处理”的清单 | 插件必须从真实文件里重新读取并核对 |
| 某个记录库不存在 | 有机会被当成空记录继续处理 | 直接停止，不会偷偷创建一个空库 |
| 记录在检查时发生变化 | 有可能拿旧结果继续判断 | 前后结果不一致就停止 |
| 从新版回退旧版 | 可以把外部记录清单作为参数传进去 | 正式入口会重新读取设置、执行记录和任务记录 |
| 检查报告 | 容易带出内部标识或路径 | 只保留不可反查的编码和数量 |

最大的变化不是“多存了一份数据”，而是迁移不再相信口头声明。它只相信插件刚刚从
硬盘上读到的事实。

## 3. 之前是什么样的？现在是什么样的？

以前，会话正文复制完成后，迁移功能还需要一份“其他记录已经有归属”的清单。测试里
经常直接把旧会话产生的清单复制给新会话。这样可以证明两份清单一样，却不能证明
Agent 执行记录和任务运行记录真的存在。

现在，流程变成了：

1. 先读取旧会话。
2. 再读取设置、Agent 执行记录和任务运行记录。
3. 一条一条确认它们能对上。
4. 全部对上才允许迁移；对不上就保留原状并报告问题。

我已经在真实插件数据的副本上跑过一次。副本运行前后没有任何文件变化，真实 Vault
也没有被修改。检查结果没有被强行做成绿色，而是继续显示阻断：

- 正式的任务运行记录库目前不存在。
- 还有 517 条旧执行信息没有找到可以负责的新记录。
- 旧会话里还有一处数据结构不符合当前迁移规则。

这说明新检查起作用了：它发现证据不够，就停下来，而不是为了完成迁移把旧声明当成
新证据。

本轮没有部署插件，没有清理历史，没有删除 Agent 会话，也没有执行自动保留或 Raw
清理。Phase 2 还差自动清理是否授权这一项；在这项决定完成前，系统只恢复已经明确
开始过的任务，不会自行发起新的删除。
