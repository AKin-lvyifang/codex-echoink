# EchoInk 知识库四步提炼协议设计

日期：2026-07-08  
更新：2026-07-09  
状态：待实施  
范围：`/check`、`/maintain`、`/reingest`、`/calibrate raw`、首页 Raw 待提炼状态

## 结论

EchoInk 的“提炼”不再等同于摘要。

新的核心协议是：

> 读懂原文 -> 拆出知识 -> 融入 Wiki / Projects -> 回写 Raw 已提炼状态

只有同时满足下面四件事，才算提炼完成：

1. `raw/` 原文、标题、路径和附件没有被改写。
2. `wiki/` 或 `projects/` 正文页出现结构化知识，而不是只有摘要。
3. 结构化知识附近保留可追溯的 `raw/` 来源证据。
4. EchoInk 插件后处理写入可信的 Raw 托管标记：`已处理`、`提炼状态`、`提炼时间`、`提炼指纹`、`提炼报告`、`提炼证据`。

`outputs/maintenance` 报告只能作为过程记录，不能单独证明提炼完成。

## 为什么要改

现在的风险不是“摘要质量不够好”，而是知识库可能出现三种假完成：

- Raw 被报告提到过，但没有进入 Wiki 结构。
- Wiki 只有一段摘要，后续搜索、复盘、问答很难复用。
- Raw 标了已处理，但找不到对应正文证据。

从第一性原理看，知识管理要解决的不是“把资料变短”，而是“让资料变成未来能被复用、追溯、更新的知识结构”。

因此提炼系统必须以 Wiki / Projects 正文为完成面，以 Raw 为证据层，以托管属性为机器状态。

## 备选方案

### 方案 A：继续增强摘要

做法：把 Raw 摘要写得更长、更细，再标记已提炼。

问题：这仍然是摘要器。它不会解决主题页重复、旧 Wiki 不更新、证据不可追溯的问题。

结论：不采用。

### 方案 B：Raw 生成独立知识卡

做法：每个 Raw 生成一个对应知识卡，例如 `wiki/sources/<title>.md`。

优点：实现简单，来源关系清楚。

问题：资料越多，知识库越碎。很多内容应该合并到已有主题页，而不是每篇文章都长出一个页面。

结论：只作为无法归类时的兜底，不作为主路径。

### 方案 C：四步提炼协议

做法：每个 Raw 先被读懂和拆解，再路由到已有 Wiki / Projects 主题页；没有合适主题页时才新建；写入结构化正文后再标记 Raw 已提炼。

优点：最符合知识库长期演进。它会减少重复页，提高复用性，也能保证“已提炼”有证据。

代价：需要重构命令职责、提示词、验证器和状态写回逻辑。

结论：采用。

## 核心对象

### Raw 来源

位置：`raw/**`

职责：保留原文、附件和来源证据。知识库命令不能改正文、标题、路径或附件。

允许 EchoInk 插件后处理写入托管元属性，但 Agent 不能直接写这些属性。

### 知识原子

提炼时从 Raw 中拆出的最小可复用知识单元。

建议类型：

- `事实`：可验证的信息、定义、数据、时间线。
- `观点`：作者判断、行业判断、产品判断。
- `方法`：可复用流程、框架、步骤。
- `案例`：具体例子、产品、公司、项目、事件。
- `风险`：限制、反例、失败条件、误区。
- `行动项`：方哥后续可以做的事。
- `问题`：值得继续研究但当前资料不能回答的点。

知识原子不是最终文件。它是提炼过程中的中间结构，用来指导写入 Wiki / Projects。

### 目标页

位置：优先 `wiki/**`，项目相关内容进入 `projects/**`。

路由规则：

- 通用概念、行业资料、方法论、长期可复用结论 -> `wiki/`
- 明确属于某个正在推进的项目、产品、插件、自动化、个人工作流 -> `projects/`
- 周报、维护报告、过程总结 -> `outputs/`
- 原始资料永远留在 `raw/`

已有主题页优先。只有找不到稳定承载页时，才新建主题页。

### 提炼证据

每个 Raw 至少要在一个目标页留下可定位证据。

有效证据必须同时满足：

- 有 `raw/` 来源链接。
- 来源链接附近有非标题、非空行、非纯元数据的实质知识内容。
- 内容进入 `wiki/**` 或 `projects/**` 的非索引正文页。

无效证据包括：

- 只出现在 `outputs/maintenance` 报告里。
- 只出现在 `raw/index.md`、`wiki/index.md` 或 `00-索引.md`。
- 只有来源链接，没有正文。
- 只有 YAML frontmatter。
- 只有代码块里的示例文本。
- 旧页面里早就存在的来源链接，但本轮没有新增知识内容。

## 命令职责

### `/check`

定位：只体检，不提炼。

它应该输出一份“提炼审计报告”，回答这些问题：

- 哪些 Raw 仍是 `Raw 待提炼`。
- 哪些 Raw 标记为已提炼，但找不到可信 Wiki / Projects 证据。
- 哪些 Wiki 页只有摘要，没有结构化知识。
- 哪些 Raw 应该并入已有主题页，而不是新建重复页。
- 哪些 Raw 指纹变化，需要重新提炼。
- 哪些历史 tracker 记录只能作为线索，不能作为机器可信状态。

写入边界：

- 只允许写 `outputs/maintenance` 体检报告。
- 不写 Raw 托管属性。
- 不写 Wiki / Projects 正文。
- 不把报告存在当成提炼完成。

### `/maintain`

定位：执行四步提炼协议。

固定流程：

1. 增量发现  
   找出 pending、changed、calibration 中需要处理的 Raw。按资料类型、更新时间和首页优先级组批。

2. 读懂原文  
   读取 Raw 正文和必要附件说明，生成内部理解结果。这个结果可以进入维护报告，但不能直接当作最终知识沉淀。

3. 拆出知识  
   把 Raw 拆成知识原子：事实、观点、方法、案例、风险、行动项、问题。

4. 路由目标页  
   先检索已有 `wiki/` 和 `projects/` 正文页，判断应该更新哪里。优先改写已有主题页；没有合适承载页才新建。

5. 融入正文  
   将知识原子改写成主题页中的结构化内容，合并重复表达，保留原有人工内容。不要简单把 Raw 摘要贴到页面底部。

6. 写来源证据  
   每个被消化的 Raw 都要在目标页新增或更新来源证据。来源证据附近必须有实质内容。

7. 机器验证  
   插件验证 Wiki / Projects 有有效证据，Raw 正文未被改，指纹匹配，报告存在。

8. 回写 Raw 状态  
   只有验证通过后，EchoInk 插件后处理才写 Raw 托管属性或非 Markdown registry。

9. 更新索引与报告  
   更新 `raw/index.md`、相关 `wiki` 索引、维护报告和 tracker。

失败规则：

- Agent 只写报告，未改 Wiki / Projects：失败，不标记 Raw。
- Agent 只加来源链接，没有实质内容：失败，不标记 Raw。
- Agent 新建重复主题页，而已有合适主题页未更新：报告风险，必要时失败。
- Raw 正文、路径、附件被改：回滚并失败。

### `/reingest`

定位：强制重新执行四步提炼。

适用场景：

- Raw 指纹变化。
- 旧提炼只是摘要，需要升级为结构化知识。
- 用户明确要求重提炼某个来源或某个目录。

行为：

- 不信任旧的摘要型产物。
- 可复用旧 Wiki 内容，但必须新增或改写结构化知识。
- 验证通过后更新 Raw 托管属性里的时间、指纹、报告和证据。

### `/calibrate raw`

定位：只校准状态，不做新提炼。

它负责把历史状态和当前证据对齐：

- Wiki / Projects 已经存在可信证据，但 Raw 未标记：补写 Raw 托管属性。
- Raw 标记已提炼，但证据缺失：降级为 `待确认` 或 `待提炼`。
- Raw 指纹变化：标记为 `待重新提炼`。
- 只有 tracker 或维护报告证据：标记为 `待校准`，不能直接算已提炼。

写入边界：

- 可以写 Raw 托管属性或 registry。
- 可以写校准报告。
- 不新增 Wiki / Projects 知识正文。

## 结构化写入规范

目标页不强制统一模板，但必须让未来的人和 Agent 能看懂。

推荐结构：

```markdown
## 关键结论

- 结论 A。来源：[[raw/articles/example]]
- 结论 B。来源：[[raw/articles/example]]

## 方法 / 框架

### 方法名

适用场景、步骤、限制。来源：[[raw/articles/example]]

## 案例

- 案例名：发生了什么，能说明什么。来源：[[raw/articles/example]]

## 风险与反例

- 风险点：为什么重要，什么情况下失效。来源：[[raw/articles/example]]

## 后续行动

- [ ] 具体行动。来源：[[raw/articles/example]]
```

写入要求：

- 不追求每页都有所有栏目。
- 不把整篇 Raw 复制进 Wiki。
- 不创建“某篇文章摘要”式孤岛页，除非它本身就是一个独立研究对象。
- 对已有主题页要做融合式改写：补充、合并、去重、调整小节，而不是尾部堆料。

## 状态模型

Raw 状态建议统一为以下几类：

- `待提炼`：没有可信证据。
- `待校准`：历史记录显示可能处理过，但缺少可信机器标记或证据不足。
- `已提炼`：指纹匹配，且有 Wiki / Projects 可信证据。
- `待重新提炼`：曾经提炼过，但 Raw 正文指纹变化。
- `提炼失败`：本轮尝试失败，保留失败原因和报告路径。

Markdown Raw 的托管属性：

```yaml
已处理: true
提炼状态: 已提炼
提炼时间: 2026-07-08T00:00:00.000Z
提炼指纹: sha256:...
提炼报告: outputs/maintenance/kb-maintenance-2026-07-08.md
提炼证据:
  - wiki/topic/example.md#关键结论
```

非 Markdown Raw 继续使用 `outputs/.raw-digest-registry.json`，字段保持同等语义。

## Agent 提示词 contract

知识库 Agent 不应该收到“总结这些 Raw”的任务，而应该收到“执行四步提炼协议”的任务。

提示词必须明确：

- Raw 是证据层，不得改写。
- 先输出内部理解，再拆知识原子。
- 必须优先检索并更新已有 Wiki / Projects 页。
- 没有合适主题页才新建。
- 每个 Raw 必须有目标页、知识内容和来源证据。
- 报告不能代替 Wiki / Projects 正文。
- 如果无法确定写入位置，应写入维护报告的“待人工判断”，不要假装完成。

## 插件验证器

插件不能只相信 Agent 说“已完成”。必须在回写状态前验证：

1. Raw 完整性  
   正文、路径、附件、权限、文件身份符合 Raw 保护规则。

2. 目标页证据  
   每个本轮 Raw 至少有一个 `wiki/**` 或 `projects/**` 非索引正文页证据。

3. 实质内容  
   来源链接附近有实质知识内容，不是只有链接、标题、frontmatter、代码块或维护报告。

4. 本轮新增或可信自愈  
   新增 Raw 必须有本轮新增正文证据。历史半提交只能按既有可信自愈规则补标。

5. 索引和报告  
   维护报告必须列出处理来源、目标页、提炼状态和失败项；索引更新失败时不能提交 Raw 已提炼状态。

## 首页和交互

首页 `Raw 待提炼` 卡片的 `提炼` 行为应该进入四步协议，而不是普通摘要。

建议：

- 点击单张 Raw 的 `提炼`：打开知识库频道并预填 `/maintain raw/articles/example.md`。
- 批量提炼：从首页筛选 `Raw 待提炼` 后触发 `/maintain 处理当前筛选 Raw`。
- 卡片状态展示：
  - `Raw 待提炼`
  - `待校准`
  - `待重新提炼`
  - `已提炼`
  - `提炼失败`

点击已提炼 Raw 时，应能看到它的 `提炼证据`，即它被写进了哪些 Wiki / Projects 页面。

## 测试验收

必须新增或更新测试覆盖：

1. `/check` 只写体检报告，不写 Raw 状态，不改 Wiki / Projects。
2. `/maintain` 处理一个新 Raw，更新已有 Wiki 页，并写 Raw 已提炼属性。
3. `/maintain` 只写报告但没改 Wiki / Projects 时失败。
4. `/maintain` 只加来源链接但没加实质内容时失败。
5. `/maintain` 遇到已有主题页时更新旧页，不新建重复页。
6. `/reingest` 能把旧摘要型产物升级为结构化知识。
7. `/calibrate raw` 能根据既有 Wiki 证据补标 Raw。
8. `/calibrate raw` 发现 Raw 已标记但证据缺失时降级状态。
9. Raw 指纹变化后状态变成 `待重新提炼`。
10. 首页 dashboard 能正确区分 `待提炼 / 待校准 / 已提炼 / 待重新提炼 / 提炼失败`。

## 实施顺序

### 第一阶段：协议与验证器

- 新增知识原子、提炼证据、Raw 状态的内部类型。
- 抽出 `digest verifier`，在状态回写前验证 Wiki / Projects 证据。
- 更新 Raw 状态判断，支持 `待重新提炼` 和 `提炼失败`。

### 第二阶段：命令职责重构

- `/check` 改为提炼审计报告。
- `/maintain` 改为四步提炼执行器。
- `/reingest` 改为强制四步提炼。
- `/calibrate raw` 改为证据校准器。

### 第三阶段：Agent Prompt 和报告格式

- 更新知识库规则模板、初始化规则和维护 prompt。
- 维护报告新增：本轮 Raw、目标页、知识类型、证据位置、失败原因。
- 报告文案避免“摘要完成”这类误导表达。

### 第四阶段：首页和状态展示

- 首页 Raw 卡片展示完整状态。
- `提炼` 动作走 `/maintain <raw path>`。
- 已提炼 Raw 可查看证据页。

### 第五阶段：真实 Vault 验收

- 在 `testing/` 中准备 Raw 和 Wiki 样例。
- 验证新 Raw -> 已有 Wiki 页 -> Raw 已提炼。
- 验证旧摘要型内容 -> 重新提炼 -> 结构化 Wiki。
- 验证失败路径不会留下假已提炼状态。

## 非目标

- 不做云端同步。
- 不把 Raw 移动、重命名或自动归档。
- 不把所有资料都拆成独立知识卡。
- 不让报告代替 Wiki / Projects 正文。
- 不要求每条知识都用统一模板，只要求可追溯、可复用、可验证。

## 已确认决策

1. `projects/` 和 `wiki/` 一样，都可以作为提炼完成面。判断标准不是目录名，而是目标页是否承载长期可复用知识。
2. 首页单张 Raw 的 `提炼` 可以直接进入可见知识库任务，并自动带上目标 Raw 路径；任务必须显示运行状态、可取消、可查看报告。批量提炼需要先展示批次清单。
3. 旧摘要型产物按激进规则处理：只要没有结构化栏目、实质知识内容和来源证据，就不能算真正提炼完成。
