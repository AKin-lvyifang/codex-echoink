# Codex EchoInk 产品白皮书

当前口径：v1.0.3  
更新时间：2026-07-10  
适用范围：Obsidian 桌面端插件

## 1. 为什么会有 EchoInk

很多 Obsidian 用户真正痛苦的地方，并不在于缺少记录入口。更常见的问题是：笔记越多，越难再用起来。

一篇文章、一段聊天、一份竞品资料、一张截图，刚收进来的时候都很有价值。但几周后再看，常见情况会变成这样：

- 原文还在，但不知道为什么当时收藏。
- 摘要写过，但没有进入长期知识结构。
- 项目资料、临时想法、输出报告混在一起。
- 想问自己的知识库，AI 给出的答案又分不清哪些来自本地资料，哪些只是模型推测。
- 想让 AI 帮忙整理，但又担心它把原始资料改坏、删掉，或者新建一堆重复页面。

EchoInk 解决的是这条链路：把收进 Obsidian 的资料，逐步变成未来能查、能引用、能继续更新的知识。

EchoInk 的产品重心，是把 Obsidian 变成一个本地知识工作台。用户仍然拥有自己的 Obsidian 仓库，EchoInk 负责把本机 AI 助手、知识库维护、写作辅助、复盘报告和安全边界组织在一起。

## 2. 它适合什么人

EchoInk 更适合已经把 Obsidian 当作长期工作区的人。

如果你只是偶尔打开 Obsidian 写几行日记，它可能显得偏重。但如果你经常收集资料、整理项目、写产品文档、做研究、跟 AI 协作，它解决的问题会更直接。

典型用户包括：

| 用户 | 主要需求 | EchoInk 的帮助 |
|---|---|---|
| Obsidian 重度用户 | 资料太多，需要持续整理 | 用首页、体检、维护报告看清知识库状态 |
| 产品经理 / 研究者 | 经常收集文章、竞品、项目资料 | 把资料沉淀到可复用的主题页，不只生成摘要 |
| 写作者 | 需要局部改写、扩写、续写、翻译 | 在编辑区生成候选内容，确认后才写入 |
| 本地 AI 用户 | 已经在用 Codex、OpenCode 或 Hermes | 在 Obsidian 内统一调用这些本机助手 |
| 长期项目维护者 | 想知道 AI 到底改了什么 | 通过过程卡片、报告、来源证据追溯结果 |

它不适合把所有事情都交给 AI 自动接管。EchoInk 的产品取向是：让 AI 做可检查的工作，让用户保留最终判断。

## 3. 核心工作模型

EchoInk 把 Obsidian 仓库分成几层。理解这几层，比记住每个按钮更重要。

| 层级 | 常见目录 | 用户可以怎么理解 |
|---|---|---|
| 原始资料层 | `raw/` | 保存原文、链接、附件和来源，像证据柜 |
| 知识层 | `wiki/`、`projects/` | 保存长期会复用的知识、方法、项目资料 |
| 输出层 | `outputs/` | 保存报告、草稿、发布稿、维护记录 |
| 收件层 | `inbox/` | 保存还没分流的想法和片段 |
| 记录层 | `journal/` | 保存日记、工作记录、周期复盘 |

### Raw 是证据柜

`raw/` 里的内容是原始资料。它可以是网页、公众号文章、PDF、手动收集的文本，也可以是附件。

EchoInk 的知识库维护任务默认保护 Raw 的正文、标题、路径和附件。原因很简单：如果原始证据被改坏，后面再漂亮的总结也失去依据。

EchoInk 可以在验证通过后给 Raw 写入“已处理、提炼状态、提炼时间、提炼报告、提炼证据”这类托管信息。托管信息是登记记录，不改写原文。

### Wiki 和 Projects 是长期知识区

真正能长期复用的内容应该进入 `wiki/` 或 `projects/`。

`wiki/` 更适合概念、方法、行业资料、长期参考。  
`projects/` 更适合正在推进的产品、插件、自动化、工作项目。

EchoInk 维护知识库时，不追求“每篇 Raw 生成一篇摘要”。它会优先把资料里的可复用知识融入已有主题页。只有找不到合适承载页时，才考虑新建页面。

### Outputs 是过程记录

`outputs/` 保存报告和阶段性产物。

报告很重要，因为它告诉用户这次做了什么、改了什么、还有什么风险。但报告本身不等于知识沉淀完成。只有内容进入 `wiki/` 或 `projects/`，并能追溯回 Raw 来源，才算真正沉淀。

## 4. 用户怎么开始用

EchoInk 安装后不会直接跑后台任务。第一次使用会先进入环境检查。

### 首次启动向导

设置页会检查本机环境，包括：

- Codex CLI 是否安装。
- Codex 是否已经登录。
- OpenCode CLI、server、模型和 Agent 是否可用。
- Hermes CLI、provider 和最小问答是否可用。

如果缺少必要条件，EchoInk 会显示缺失项、安装命令、复制按钮和文档入口。

`Start` 按钮只做两件事：记录用户已经完成启动向导，并打开 EchoInk 侧栏。它不会自动发送消息，不会自动初始化知识库，也不会自动维护 Vault。

### 四个日常入口

EchoInk 的日常使用可以从四个入口开始。

| 入口 | 看见什么 | 适合做什么 |
|---|---|---|
| Agent 侧栏 | 会话、附件、模型、权限、过程卡片 | 让本机 AI 助手处理文件任务 |
| 首页工作台 | 健康分、Raw 待提炼、日历、热力图、笔记卡片 | 每天看知识库整体状态 |
| 知识库频道 | `/ask`、`/check`、`/maintain` 等命令 | 管理当前 Obsidian 仓库 |
| 编辑区命令 | 改写、扩写、续写、翻译 | 对选中文字生成候选内容 |

普通 Agent 会话必须先选择一个本地文件夹作为工作区。这个设计是为了让 AI 的读写范围明确下来。附加一篇笔记、图片或文件，只代表本轮上下文，不代表把整个 Obsidian 仓库交给 AI。

知识库频道不同。它默认服务当前打开的 Obsidian 仓库，用来处理 Raw、Wiki、Projects、Outputs、Inbox 和 Journal。

## 5. 知识库能力

知识库频道是 EchoInk 的核心。用户既可以输入明确命令，也可以用自然语言说“只体检一下”“维护知识库”“收集这个链接”。

EchoInk 不会把所有自然语言都强行当成知识库命令。普通聊天仍然可以按普通 Agent 对话处理，避免误触发大范围知识库操作。

### 初始化

`/init` 用来初始化知识库结构。

它先生成预览，不直接写文件。用户确认后输入 `/init confirm`，才会创建目录和规则文件。

这个流程适合新 Vault。它会准备 Raw、Wiki、Projects、Outputs、Inbox、Journal、Templates、Assets、Archive 等基础结构。已有文件不会被删除或覆盖。

### 收集

EchoInk 支持把资料先收进 Raw 或 Inbox。

| 收集内容 | 默认落点 |
|---|---|
| 手动想法 | `inbox/` |
| 手动文本资料 | `raw/articles/手动收集/` |
| 微信公众号链接 | `raw/articles/微信公众号/` |
| 普通网页链接 | `raw/articles/网页收藏/` |
| PDF / DOCX / Markdown / TXT | 文本进 `raw/articles/文件收藏/`，附件进 `raw/attachments/` |
| 当前图片或 PDF | `raw/attachments/` |

遇到网页登录、验证码、平台限制时，EchoInk 会报错或保存失败原因，不绕过验证。

跨端快捷收集网关还没有作为当前版本发布能力。微信、飞书、Chrome 扩展、iOS Shortcuts、安卓分享统一进入云端 Relay，再由家里电脑写入 Obsidian，这套方案目前是 PRD 阶段。

### 提问

`/ask` 是只读问答。

它会先检索本地 `wiki/`，再把 `journal/`、`outputs/` 作为背景依据。回答时需要区分：

- 哪些内容来自当前 Vault。
- 哪些只是 AI 的补充信息或推断。
- 如果找不到本地依据，也要说明没有找到。

`/ask` 不写文件。

### 体检

`/check` 只做体检，不做提炼。

它适合先看清楚问题：

- Raw 还有多少待提炼。
- Wiki 是否存在断链。
- Inbox 是否积压。
- 最近维护有没有失败。
- 索引、tracker、Raw 状态是否可能漂移。

`/check` 可以写体检报告，但不应该改 Raw、Wiki 或 Projects 正文。

### 维护

`/maintain` 执行知识库维护，是 EchoInk 把资料变成知识的主命令。

它的目标是把 Raw 中有长期价值的内容融入 `wiki/` 或 `projects/`。维护完成后，用户应该能在主题页里看到结构化知识，也能看到来源来自哪篇 Raw。

如果任务只写了维护报告，没有写入 Wiki 或 Projects，就不能算真正完成提炼。

### 重新提炼和状态校准

`/reingest` 用来强制重新提炼。适合 Raw 内容变化、旧页面只有摘要、用户明确指定某篇资料重做的场景。

`/calibrate raw` 用来校准 Raw 状态。它只对齐“状态”和“证据”，不调用 Agent 重新提炼。

典型情况是：

- Wiki / Projects 已经有可信证据，但 Raw 没标记。
- Raw 标记已处理，但找不到证据。
- Raw 内容变了，需要标记为待重新提炼。

### 处理 Outputs 和 Inbox

`/outputs` 用来处理输出区。它会从报告、草稿、阶段性产物里提炼长期价值，适合把临时成果变成长期知识。

`/inbox` 用来整理收件箱。它适合处理临时想法、零散片段和待分流输入。需要移动、删除、归档时，EchoInk 应该先在报告里说明。

### 日记、周报和历史

`/journal` 会根据当天真实工作记录生成或更新工作日记。默认路径类似：

```text
journal/daily/YYYY-MM/YYYY-MM-DD-周X.md
```

`/week` 可以生成知识库周报，也可以生成 Agent 对话周报。输出包括 Markdown 文件和 HTML 看板。复盘自动任务在设置里默认关闭，当前白皮书把手动和命令生成作为稳定使用方式。

`/history` 用来查看知识库频道历史。旧消息会按天保存在插件本地历史目录，避免所有历史都堆在当前页面。

`/clear` 只清空当前页面可见消息，不删除本地历史。  
`/cancel` 用来取消正在运行的知识库任务，并尽量把状态记录为已取消。

## 6. 四步提炼和 Raw 保护

EchoInk 对“提炼”的定义很严格。

真正完成提炼，要同时满足四件事：

1. Raw 原文、标题、路径和附件没有被改坏。
2. Wiki 或 Projects 中出现结构化知识。
3. 这些知识旁边能追溯到 Raw 来源。
4. EchoInk 写下可信状态和报告路径。

这套流程可以概括为：

```text
读懂原文 -> 拆出知识 -> 融入 Wiki / Projects -> 回写 Raw 状态
```

举个例子。

如果 Raw 是一篇关于产品定价的文章，低质量结果只是生成一段“本文介绍了定价策略”的摘要。

EchoInk 更希望得到这样的结果：

- 把“免费试用”“分层套餐”“价格锚点”等方法写入产品定价主题页。
- 把文章里的案例放到对应案例小节。
- 标明来源是哪个 Raw 文件。
- 验证证据存在后，再把 Raw 标记为已提炼。

未来用户问“产品定价有哪些常见方法”时，知识库可以直接复用主题页，而不是重新翻一堆原文摘要。

### 什么时候会失败

EchoInk 不应该因为 Agent 说“完成了”就相信任务完成。

这些情况都应该被视为失败或需要复核：

- 只写了报告，没有写入 Wiki 或 Projects。
- 只加了来源链接，没有写入实质知识。
- 已有合适主题页，却新建了重复页面。
- Raw 正文、路径或附件被意外改动。
- Raw 标记已处理，但找不到可信证据。

这个设计会让维护流程更慢一点，但能减少“看起来整理了，实际不可用”的假完成。

## 7. Agent 工作区

EchoInk 也提供普通 Agent 会话，用来处理知识库之外的文件任务。

用户可以在侧栏里：

- 新建会话。
- 选择本地文件夹作为工作区。
- 附加当前笔记、文件或图片。
- 选择模型、思考强度、速度和权限。
- 在 Plan 模式里先做计划。
- 在 Agent 模式里执行任务。
- 查看命令、文件修改、工具调用和最终结果。

### 可切换的本机助手

EchoInk 当前支持三类 Agent 后端。这里的“后端”，就是负责执行任务的本机助手。

| 后端 | 当前定位 | 说明 |
|---|---|---|
| Codex CLI | 主力路径 | 过程时间线最完整，适合普通 Agent 工作和文件任务 |
| OpenCode | 已接入 | 可用于聊天、知识库和写作等能力，过程展示更简化 |
| Hermes | 第一阶段已接入 | 支持 Hermes profile、provider、CLI/API 体系，管家化能力仍在路线图 |

EchoInk 的长期方向是成为 Obsidian 里的统一工作台。它自己负责知识库、安全、资源和界面，不同助手负责执行。

### 过程时间线和任务队列

在 Codex 路径下，EchoInk 可以把 AI 的过程渲染成卡片，例如思考、计划、命令、文件修改、工具调用、上下文用量、失败原因。

OpenCode 和 Hermes 在没有同等公开事件能力时，会显示更简化的运行状态。EchoInk 不把简化状态包装成 Codex 级别的直播过程。

当一个任务还在运行时，用户可以继续输入后续任务并加入队列。队列会锁定当时的文本、附件、模型、权限、模式和工作区。当前任务成功后才执行下一条；如果失败或被停止，队列会暂停，等待用户手动继续。

知识库命令也会串行执行，避免多个任务同时抢同一个 Vault。

## 8. 首页工作台

EchoInk 首页是知识库的总览面板，不是欢迎页。

它会展示：

- 当前 Vault 名称。
- 知识库健康状态和健康分。
- Raw 待提炼数量。
- Wiki、Raw、Inbox、Outputs 等数量。
- 年度体检热力图。
- 知识活动日历。
- 最近更新或推荐查看的笔记卡片。
- 最近维护报告和任务状态。

首页的价值是让用户每天都能知道知识库现在处在什么状态。

比如 Raw 待提炼变多，说明资料正在堆积。健康分下降，说明可能有断链、孤儿页、过时草稿或未处理来源。最近维护报告可以告诉用户上次整理做了什么，哪里还需要复核。

笔记卡片支持筛选和排序，也可以复制 Obsidian 内链、相对路径和 Markdown 链接。

当前本地已有“知识库健康能量格”视觉原型，用来优化健康指标展示。它仍属于原型方向，不能写成已发布正式界面。

## 9. 写作辅助和复盘

### 写作辅助

EchoInk 支持在编辑区对选中文字执行：

- 改写。
- 扩写。
- 续写。
- 翻译成英文。

它不会直接替换原文。AI 生成内容后，会先作为灰色候选出现在编辑区：

- 按 Enter 接受候选。
- 按 Esc 取消候选。
- 切换文件或正文变化时自动取消。

写作辅助有快速、质量、严格三种模式。它仍属于实验能力，默认关闭，适合先在非关键笔记中试用。

### 复盘周报

EchoInk 可以生成两类周报：

| 周报 | 关注什么 |
|---|---|
| 知识库周报 | 本周知识库命令、健康分、维护报告、失败和风险信号 |
| Agent 对话周报 | 普通会话数量、用户消息、上下文压缩、失败中断、提示词样本 |

周报会输出 Markdown 和 HTML。Markdown 适合继续在 Obsidian 里编辑和归档，HTML 看板适合快速浏览。

HTML 预览只允许打开 EchoInk 生成的周报 HTML，避免随意打开不相关本地网页。

## 10. 资源、权限和隐私

EchoInk 是本地优先的桌面插件，但本地优先不等于完全离线。

Codex、OpenCode、Hermes 或自定义模型服务可能会把 prompt、附件或文件上下文发送给对应服务。具体发送到哪里，取决于用户配置的后端和模型。

EchoInk 的边界是：

- 只支持 Obsidian 桌面端。
- 不默认上传整个 Vault。
- 普通会话必须先选择工作区。
- 附件只作为本轮上下文。
- 自定义 API key 保存在本机 Obsidian 插件数据中。
- 不静默安装 Codex、OpenCode、Hermes。
- 不绕过网页登录、验证码或平台限制。
- 知识库维护中保护 Raw 原始资料。

### 权限模式

| 权限 | 用户理解 |
|---|---|
| 只读 | AI 只能看，不能写文件 |
| 工作区可写 | AI 可以改用户选择的工作区 |
| 更高权限 | AI 拥有更大的本机操作能力，风险更高 |

普通 Agent 会话以用户选择的工作区和权限为准。知识库维护使用更窄的写入范围，并由 EchoInk 负责 Raw 保护、状态校验和报告。

### Skills 和 MCP

EchoInk 支持把外部能力接入当前 Vault。

`Skill` 是一份可复用的工作说明，例如“按某种格式写报告”。  
`MCP` 是外部工具连接，让 AI 有机会调用某些工具。

EchoInk 的资源管理原则是：

- 资源开关只影响当前 Obsidian Vault。
- 不直接修改 Codex、OpenCode、Hermes 的全局配置。
- 仅导入的 MCP 不等于可调用。
- 只有补齐连接配置、审批和日志后，才应被视为可执行工具。

当前版本已有 EchoInk MCP broker 基础层，支持有明确连接配置的 MCP 列工具、调用工具、审批和记录日志。更完整的 Agent 工具闭环，以及 OpenCode / Hermes 的完整 rich stream，仍属于后续增强方向。

## 12. 未来方向

### 跨端快捷收集

目标是让微信、飞书、Chrome、iOS、安卓都能把资料送进同一个入口。

规划结构是：

```text
外部入口
-> 云端 Relay 收件箱
-> 家里电脑 Home Agent 拉取
-> EchoInk 插件写入 Obsidian
-> 后续 /maintain 消化
```

第一版原则仍然是：云端只收件和排队，不直接写本地 Vault。

### 更完整的工具闭环

EchoInk 需要继续把工具列表、用户审批、工具调用、调用日志和失败展示收束到插件本体。

这条路线的重点，是让 EchoInk 自己负责当前 Vault 的工具边界，而不是直接照搬某个后端的全局工具配置。

### Hermes 管家化

Hermes 的 profile、memory、dream、insights 等能力，未来可以让 EchoInk 更像知识库管家。

但它必须遵守同一套边界：

- 先给建议。
- 再生成草稿。
- 需要用户确认后才写入。
- 不能绕过 Raw 保护、报告和审批。

### 更清楚的维护结果展示

维护结果不应该长期停留在大段 Markdown 里。更好的体验是运行卡片和结果卡片：用户一眼看到当前在做什么、哪些资料完成了、哪些需要复核、证据和报告在哪里。

这部分已经有本地 HTML 原型和实现探索，最终仍以真实 Obsidian 插件验收为准。

## 13. 怎么判断是否该采用

可以用三个问题判断 EchoInk 是否适合当前项目。

第一，你是否已经把 Obsidian 当作长期知识库，而不只是临时草稿本？

第二，你是否经常把资料收进来之后，又需要在几天或几周后重新找、重新问、重新引用？

第三，你是否愿意让 AI 参与整理，但要求它留下报告、来源、边界和可检查结果？

如果三个答案大多是“是”，EchoInk 的价值会比较明显。它会帮助你把“资料堆积”变成“知识维护”。

如果只是想要一个纯聊天机器人，或者希望 AI 自动接管整个电脑和整个 Vault，EchoInk 当前不是这个方向。

## 15. 对外介绍短版

Codex EchoInk 是一个 Obsidian 桌面端插件，用来把本机 AI 助手接入个人知识库。它支持 Agent 工作区、知识库首页、Raw 资料收集、知识库问答、体检维护、四步提炼、写作辅助和周报复盘。它强调本地优先、用户授权、Raw 保护和来源可追溯，目标是把越积越多的资料逐步变成未来能查、能用、能更新的知识结构。
